class HybridChimeraModel(nn.Module):
    def __init__(self, base_model, layer_plan):
        super().__init__()
        self.base = base_model
        self.layer_plan = layer_plan

    def forward(self, input_ids, attention_mask=None):
        x = self.base.embed_tokens(input_ids)

        for i, layer in enumerate(self.base.layers):
            plan = self.layer_plan[i]
            x = layer(x, attention_mask=attention_mask, plan=plan)

def distill_loss(student_logits, teacher_logits, temperature=2.0):
    t = temperature
    p_t = torch.softmax(teacher_logits / t, dim=-1)
    log_p_s = torch.log_softmax(student_logits / t, dim=-1)
    return torch.nn.functional.kl_div(log_p_s, p_t, reduction="batchmean") * (t * t)

That is where I would spend GPU time only after the forward path is stable, because your doc makes distillation the proof that gradients flow through the hybrid architecture and that the model actually improves after training.  

A minimal training step would look like this:

def train_step(student, teacher, batch, optimizer):
    student.train()
    teacher.eval()

    with torch.no_grad():
        t_logits = teacher(**batch).logits

    s_logits = student(**batch).logits
    loss = distill_loss(s_logits, t_logits, temperature=2.0)

    optimizer.zero_grad(set_to_none=True)
    loss.backward()
    torch.nn.utils.clip_grad_norm_(student.parameters(), 1.0)
    optimizer.step()

    return float(loss.item())

class FallbackMamba(nn.Module):
    def __init__(self, d_model):
        super().__init__()
        self.norm = nn.LayerNorm(d_model)

        self.in_proj_x = nn.Linear(d_model, d_model, bias=False)
        self.in_proj_b = nn.Linear(d_model, d_model, bias=False)
        self.out_proj_c = nn.Linear(d_model, d_model, bias=False)

import torch.nn.functional as F

def distill_loss(student_logits, teacher_logits, labels, T=2.0, alpha=0.5):
    # Normalize logits (critical)
    student_logits = student_logits / (student_logits.std(dim=-1, keepdim=True) + 1e-6)
    teacher_logits = teacher_logits / (teacher_logits.std(dim=-1, keepdim=True) + 1e-6)

    loss_ce = F.cross_entropy(
        student_logits.view(-1, student_logits.size(-1)),
        labels.view(-1)
    )

    p_s = F.log_softmax(student_logits / T, dim=-1)
    p_t = F.softmax(teacher_logits / T, dim=-1)

    loss_kl = F.kl_div(p_s, p_t, reduction='batchmean') * (T ** 2)

    return (1 - alpha) * loss_ce + alpha * loss_kl

🧠 1. True Mamba Math (What you’re actually trying to implement)

✅ Continuous-time SSM (ground truth)

A Mamba-style layer is based on a state space model (SSM):

\frac{d s(t)}{dt} = A s(t) + B x(t)
y(t) = C s(t)

Where:
	•	s(t) \in \mathbb{R}^{d_{state}}
	•	x(t), y(t) \in \mathbb{R}^{d_{model}}
	•	A \in \mathbb{R}^{d_{state} \times d_{state}}
	•	B \in \mathbb{R}^{d_{state} \times d_{model}}
	•	C \in \mathbb{R}^{d_{model} \times d_{state}}

⸻

✅ Discretized (what you actually run)

With timestep Δ:

s_t = \bar{A} s_{t-1} + \bar{B} x_t
y_t = C s_t

Where:
	•	\bar{A} = e^{A \Delta}
	•	\bar{B} = \int_0^\Delta e^{A\tau} B d\tau

⸻

✅ What Mamba actually does (key simplification)

Mamba does structured parameterization:

Instead of full A, it uses:
	•	diagonal or low-rank structure
	•	often represented as:
A = \text{diag}(\lambda)

So:

s_t[i] = \lambda_i s_{t-1}[i] + b_i^\top x_t

This reduces compute from O(d^2) → O(d)

✅ Minimal correct PyTorch version

This is the first “real” Mamba core you should implement:

class SimpleSSM(nn.Module):
    def __init__(self, d_model, d_state):
        super().__init__()
        self.d_state = d_state

        # A diagonal
        self.A = nn.Parameter(torch.randn(d_state))  # λ

        # B and C projections
        self.B = nn.Linear(d_model, d_state, bias=False)
        self.C = nn.Linear(d_state, d_model, bias=False)

    def forward(self, x):
        # x: (B, T, d_model)
        B, T, D = x.shape

        s = torch.zeros(B, self.d_state, device=x.device)
        outputs = []

        for t in range(T):
            x_t = x[:, t]

            s = torch.exp(self.A) * s + self.B(x_t)
            y = self.C(s)

            outputs.append(y)

        return torch.stack(outputs, dim=1)

👉 For Ouroboros:

	•	use mamba-ssm for real training

With a d_state gradient, I would not try to treat “the SSM state” as one uniform tensor. I would treat it as a per-layer memory budget that changes with depth, exactly like your spec’s 18-layer gradient from d16 up to d256.  

The main idea

Attention and SSM still stay separate:
	•	Attention layers keep a normal KV cache.
	•	SSM layers keep a state vector whose length depends on that layer’s d_state.

So the only real change from a uniform hybrid is this:

s_\ell \in \mathbb{R}^{d_{\text{state}}(\ell)}

instead of one shared state size for all SSM layers.

If layer 3 uses d_state=24 and layer 17 uses d_state=256, then layer 3 stores a 24-wide state and layer 17 stores a 256-wide state. The hidden size seen by the rest of the model can still stay constant.

⸻

What that means in inference

You keep two independent memory systems:

\text{KV cache} = \{K_\ell, V_\ell\}_{\ell \in \mathcal{A}}

\text{SSM state} = \{s_\ell\}_{\ell \in \mathcal{S}}

where \mathcal{A} is the set of attention layers and \mathcal{S} is the set of SSM layers.

For a gradient model, the SSM state dictionary is heterogeneous:

ssm_state = {
    1: torch.zeros(batch, 16, device=device),
    2: torch.zeros(batch, 16, device=device),
    3: torch.zeros(batch, 24, device=device),
    4: torch.zeros(batch, 24, device=device),
    5: torch.zeros(batch, 32, device=device),
    # ...
    17: torch.zeros(batch, 256, device=device),
    18: torch.zeros(batch, 256, device=device),
}

That is the simplest correct representation.

⸻

The math of the gradient

A clean way to write your depth schedule is:

d_{\text{state}}(\ell)=\operatorname{round}\left(d_{\min}\left(\frac{d_{\max}}{d_{\min}}\right)^{\ell/(L-1)}\right)

where:
	•	L = number of layers
	•	d_{\min}=16
	•	d_{\max}=256

That gives a monotonic growth in memory capacity with depth.

For your 18-layer example, that matches the spirit of the stepwise schedule you wrote in the spec, where shallow layers handle local token patterns and deeper layers handle broader structure.  

⸻

How I would wire the actual SSM layer

The important part is that the input and output interfaces stay fixed, while only the internal state changes size.

Let x_\ell(t) \in \mathbb{R}^{d_{\text{model}}} be the layer input at token t. Then:

u_\ell(t)=W^{(\ell)}_{\text{in}}x_\ell(t)

s_\ell(t)=A_\ell s_\ell(t-1)+B_\ell u_\ell(t)

y_\ell(t)=W^{(\ell)}_{\text{out}}s_\ell(t)

where:
	•	s_\ell(t) \in \mathbb{R}^{d_{\text{state}}(\ell)}
	•	B_\ell \in \mathbb{R}^{d_{\text{state}}(\ell)\times d_{\text{model}}}
	•	W^{(\ell)}_{\text{out}} \in \mathbb{R}^{d_{\text{model}}\times d_{\text{state}}(\ell)}

So the layer boundary always speaks d_model, but the internal state can vary by depth.

⸻

What changes for attention layers

Almost nothing, structurally.

Attention layers still use:
	•	the same model width,
	•	the same head dimension,
	•	the same KV cache mechanics.

So your hybrid becomes:

Layer 0: attention, KV cache
Layer 1: SSM, state size 16
Layer 2: SSM, state size 16
Layer 3: SSM, state size 24
...
Layer 17: SSM, state size 256

The attention side does not care that the neighboring SSM layer has a different d_state. The two systems only meet through the hidden activations passed between layers.

⸻

How I would make the gradient actually useful

The risk in a d_state gradient is that it becomes just “bigger state later” without real specialization. To prevent that, I would tie the gradient to the role of the layer.

A practical rule:
	•	small d_state layers learn local compression,
	•	medium d_state layers learn module-level summaries,
	•	large d_state layers learn long-range structural memory.

That matches your spec’s intent: shallow layers for token patterns, deep layers for architecture-level recall.  

I would also couple the decay parameter to depth:

\alpha_\ell=\sigma(\beta \cdot \text{depth}(\ell)+\gamma)

so deeper layers can retain information differently from shallow ones.

⸻

The real implementation issue: ragged state buffers

The tricky part is batching and kernels.

Option 1: simplest

Allocate one tensor per layer, sized exactly to that layer:

layer_state[layer_idx] = torch.zeros(batch, d_state[layer_idx], device=device)

This is easiest to debug.

Option 2: faster for kernels

Group layers with the same d_state into buckets, so you run fewer kernel variants.

Example buckets:
	•	d16 for layers 1–2
	•	d24 for layers 3–4
	•	d32 for layers 5–6
	•	etc.

That helps if you later write fused CUDA kernels.

Option 3: single padded buffer

Store everything in a padded max_d_state tensor and mask unused parts.

This is convenient for code simplicity, but wastes memory and can hide bugs. I would only use it for early experiments.

⸻

How I would connect this to your β experiment

Your gradient experiment becomes more meaningful if β is allowed to shape the memory curve, not just gate a single layer. In other words:

d_{\text{state}}(\ell) \text{ sets capacity}

\beta \text{ sets retention behavior}

That gives you two separate axes:
	1.	capacity — how much memory a layer can hold,
	2.	timescale — how long it holds it.

That is much better than making β do everything.

A concrete implementation sketch

class GradientSSMLayer(nn.Module):
    def __init__(self, d_model, d_state, layer_idx, total_layers):
        super().__init__()
        self.d_state = d_state
        self.depth = layer_idx / (total_layers - 1)

        self.in_proj = nn.Linear(d_model, d_state, bias=False)
        self.out_proj = nn.Linear(d_state, d_model, bias=False)

        self.A = nn.Parameter(torch.zeros(d_state))
        self.beta = nn.Parameter(torch.zeros(1))
        self.gamma = nn.Parameter(torch.zeros(1))

    def init_state(self, batch_size, device):
        return torch.zeros(batch_size, self.d_state, device=device)

    def forward(self, x, state):
        # x: (B, D)
        alpha = torch.sigmoid(self.beta * self.depth + self.gamma)
        decay = torch.exp(self.A * alpha)

        u = self.in_proj(x)
        state = decay * state + u
        y = self.out_proj(state)

        return y, state

That is the core pattern I would use.

How this affects your KV cache / SSM split

With a d_state gradient, I would make attention slightly more sparse toward deeper layers, because the deeper SSM layers have more memory capacity and can absorb more of the long-range burden. The effect is:
	•	shallow attention layers do exact local routing,
	•	deeper SSM layers do larger-scale compression and retention,
	•	deep attention layers only appear where exact retrieval is still worth the cost.

That matches the spirit of your hybrid plan and keeps the architecture from becoming “attention everywhere plus huge SSM everywhere.”
